# S0 — Scaffold

The prerequisite slice: a runnable, deployable monolith skeleton with the DDD layering,
persistence, and test harness in place — but **no domain features yet**. Every later slice
(S1–S14) is built test-first inside this structure.

## Definition of done

1. One process: a **Hono** server serves both the BFF API and the built **Svelte** app.
2. `dev`, `build`, `start`, `test`, and Drizzle migration commands all work.
3. **Drizzle + SQLite** connected; an initial migration creates the `artefact` and
   `data_entry` tables; `drizzle-kit` generate/migrate wired. *(BetterAuth's user/session/
   account tables are generated by BetterAuth's CLI when auth is wired in **S1**, not here —
   see Implementation note.)*
4. **Tailwind + shadcn-svelte** installed and rendering (one demo component).
5. The **domain layer is pure TypeScript** (no Hono, no Drizzle imports) and unit-tested
   with **Vitest** — a trivial passing test proves the harness.
6. **Filesystem payload store** root configured (env-driven), with a tiny adapter stub.
7. A **multi-stage Dockerfile** builds a single image that runs the server; documented for
   **Coolify** deploy.
8. `health` endpoint returns 200.

No `Artefact`/auth behavior is implemented here — just the skeleton those slices fill in.

## Architecture decision to confirm (the one real fork)

The chosen server is **Hono as BFF**. That makes the cleanest fit:

- **Recommended: Hono server + a Vite-built Svelte SPA (client-side routing).** Hono owns
  *all* server concerns (BFF API, auth mount, artefact serving, static asset serving). The
  Svelte app is a pure client bundle Hono serves. This keeps a single server and a single
  deployable, and the BFF boundary stays explicit. Trade-off: **shadcn-svelte** targets
  SvelteKit, so in a plain Vite+Svelte SPA its CLI/registry setup is manual (the components
  themselves are just Svelte + Tailwind and work fine).
- **Alternative: SvelteKit (node adapter) fronted/augmented by Hono.** Gets first-class
  shadcn-svelte, but SvelteKit's own server endpoints overlap the BFF role and muddy "Hono
  is the BFF." More moving parts for a monolith.

**This spec assumes the recommended option** (Hono + Vite/Svelte SPA). Confirm before
scaffolding — it shapes the whole `client/` tree.

**Package manager:** assume **pnpm** (swap to npm if preferred — only affects command
prefixes below).

```
      index.ts                   # entry: create app, listen
      app.ts                     # Hono app, middleware, route mounts, static serving
      auth.ts                    # BetterAuth instance (Drizzle adapter) — wired in S1
      routes/                    # one module per slice (filled S1+)
      runtime/                   # served-artefact bootstrap (localStorage hijack) — S13
      env.ts                     # parsed/validated config
    client/                      # Svelte SPA (Vite)
      main.ts
      App.svelte
      app.css                    # Tailwind entry
      lib/
        components/ui/           # shadcn-svelte components
        api/                     # typed BFF client (consumes shared contracts)
      routes/                    # client-side pages (filled S2+)

  data/                          # runtime (gitignored): sqlite db file + payload store root
    artefactor.db
    payloads/
```

**Why this shape:** the `domain/` layer encodes DDD aggregates and invariants with **no
infrastructure dependencies**, so TDD tests them against in-memory repositories — fast and
framework-free. `infra/` adapts those ports to Drizzle/filesystem. `server/` is the
composition root wiring BFF routes → domain → adapters. This is the structure every later
slice plugs into.

## Persistence (Drizzle + SQLite)

Initial `schema.ts` (domain tables; BetterAuth contributes its own via its Drizzle schema):

- **`artefact`**: `id` (text/uuid, pk), `owner_id` (text → user), `title` (text),
  `kind` (text enum), `visibility` (text enum: private|authenticated|public),
  `public_slug` (text, unique, nullable), `status` (text enum: active|archived),
  `payload_ref` (text — filesystem key), `payload_bytes` (integer),
  `payload_hash` (text), `created_at`, `updated_at`, `archived_at` (nullable).
  - Unique index on `public_slug`. Index on `owner_id`. Index on `(status, visibility)` for
    the gallery.
- **`data_entry`**: `id` (pk), `artefact_id` (text → artefact, fk),
  `author_id` (text → user), `blob` (text — JSON), `created_at`, `updated_at`.
  - **Unique index on `(artefact_id, author_id)`** (enforces one-per-pair / upsert).
- **BetterAuth tables**: provided by BetterAuth's Drizzle schema (user, session, account,
  verification, plus the api-key plugin's table). Generated into the same SQLite DB and the
  same migration pipeline.

**The 100 MB HTML payload is NOT a column** — only `payload_ref` + size + hash live in the
row; bytes live on the filesystem (see below).

Commands: `drizzle-kit generate` (from schema) → `drizzle-kit migrate` (apply). SQLite via
`better-sqlite3` (sync, simplest for a single-container monolith); enable WAL.

## Filesystem payload store

- Root from env `ARTEFACTOR_PAYLOAD_DIR` (default `./data/payloads`), a Docker **volume** in
  prod so payloads survive redeploys.
- Adapter API (port lives in domain as a storage interface; impl in `infra/storage`):
  `put(bytes) -> { ref, bytes, hash }`, `get(ref) -> stream/bytes`, `delete(ref)`.
- `ref` is content-addressed or uuid-named; never the user-supplied filename.
- S0 ships the adapter stub + interface; real wiring lands with S2.

## Server wiring (Hono)

- `app.ts`: Hono app with logger + error middleware; mounts:
  - `/api/*` → BFF routes (added per slice).
  - BetterAuth handler route (S1).
  - artefact serving route by slug (S6) — also where the localStorage-hijack bootstrap is
    injected (S13).
  - static: serve the built Svelte client (`dist/client`) with SPA fallback to `index.html`.
  - `GET /health` → 200.
- `env.ts`: validate config (DB url, payload dir, auth secret, base url) at boot; fail fast.
- Single listen port (env `PORT`, default 3000).

## Dev / build / run

Intended scripts (these become the CLAUDE.md **Commands** section once implemented):

| Command | Purpose |
|---------|---------|
| `pnpm dev` | Vite dev server for the client + Hono in watch — single dev experience |
| `pnpm build` | `vite build` (client → `dist/client`) + bundle server (`dist/server`) |
| `pnpm start` | run `dist/server` (serves API + built client) |
| `pnpm test` | Vitest (unit, domain-first) |
| `pnpm test <file> -t "name"` | run a single test by name |
| `pnpm db:generate` | drizzle-kit generate migration from schema |
| `pnpm db:migrate` | apply migrations |
| `pnpm db:studio` | drizzle studio |
| `pnpm check` | `svelte-check` + `tsc --noEmit` |
| `pnpm lint` / `pnpm format` | eslint / prettier |

Dev integration: run the Hono server and Vite together (e.g. `@hono/vite-dev-server`, or
Vite for the client + `tsx watch` for the server with the client proxied). Exact mechanism
is an implementation detail of S0; the DoD is "`pnpm dev` gives hot client + reloading API."

## Docker / Coolify

- **Multi-stage Dockerfile**: deps → build (`pnpm build`) → slim runtime image running
  `node dist/server` (or the bundled entry). Single image, single process.
- Migrations run on container start (entrypoint runs `db:migrate` then starts the server),
  so a fresh volume self-initializes.
- **Volumes** (declared in Coolify): the SQLite DB file and the payload dir must be on a
  persistent volume — they are the entire state of the monolith.
- Env via Coolify: `PORT`, `ARTEFACTOR_PAYLOAD_DIR`, DB path, BetterAuth secret + base URL,
  (later) Google OAuth creds.
- Expose `PORT`; Coolify terminates TLS and reverse-proxies.

## Testing harness (TDD)

- **Vitest** for unit tests, co-located `*.test.ts` next to domain modules.
- Domain tests use **in-memory repository** implementations of the ports — no DB needed —
  so invariant tests are fast and pure. This is the primary TDD surface for S1–S14.
- A thin integration layer (real Drizzle against a temp SQLite file) is added when the first
  adapter lands (S2); S0 only needs the runner proven with one trivial domain test.

## Acceptance criteria

- [ ] `pnpm install && pnpm build && pnpm start` serves the app and `GET /health` → 200.
- [ ] `pnpm dev` runs client + API together with reload.
- [ ] `pnpm db:generate && pnpm db:migrate` creates `artefact` and `data_entry` in SQLite.
      (BetterAuth tables follow in S1.)
- [ ] A demo shadcn-svelte component renders with Tailwind styles.
- [ ] `pnpm test` runs Vitest green on one trivial domain test using an in-memory repo.
- [ ] `docker build` produces an image that boots, runs migrations, and serves `/health`.
- [ ] Payload dir and DB path are env-driven and documented in `.env.example`.

## Decided

- **Svelte topology: Hono + a Vite-built Svelte SPA.** Hono is the single server and the
  only BFF; the Svelte app is a client bundle Hono serves. shadcn-svelte is set up in
  non-Kit (manual) mode.
- **Package manager: pnpm.**
- **SQLite driver: `better-sqlite3`** (sync, single-container friendly; WAL enabled).

## Implementation notes (from building S0)

- **Stack landed on:** Node 24, pnpm 11, Hono 4 + `@hono/node-server`, Svelte 5, Vite 8,
  Tailwind v4 (`@tailwindcss/vite`, CSS-first — no `tailwind.config.ts`), Drizzle 0.45 +
  `drizzle-kit` 0.31, Vitest 4, zod 4.
- **Server build via esbuild**, not `tsc`: `src/server/index.ts` and `migrate.ts` are
  bundled (`--format=esm --packages=external`) to `dist/server/*.js`. Avoids ESM
  `.js`-extension friction; `pnpm check` (svelte-check + `tsc --noEmit`) does type safety.
- **Native build approval:** pnpm 11 blocks dependency build scripts by default, and
  `onlyBuiltDependencies` in `pnpm-workspace.yaml` did **not** auto-build on install in
  11.9. The reliable non-interactive trigger is **`pnpm approve-builds --all`** after
  install — used in the Dockerfile and required locally for `better-sqlite3` (no prebuilt
  exists for Node 24 / ABI 137, so it compiles from source; the build stage installs
  `python3 make g++`).
- **shadcn-svelte:** the CLI targets SvelteKit, so the demo `Button` was added manually
  (tailwind-variants + a `cn` util); `components.json` is present for future CLI use.
- **Tailwind tokens:** a minimal zinc token set lives in `app.css` via `@theme inline`; the
  shadcn CLI would expand these later.
